## 2. 지표 규칙 (확정)

### 2.1 지표 정의 (단일 소스)
| 키 | 라벨 | 초기값 | 방향 | 실패조건 |
|---|---|---:|---|---|
| `mental` | 멘탈 | 70 | 높을수록 좋음 | `<= 0` |
| `progress` | 오늘의 진척 | **40** (기존 30→상향, D7) | 높을수록 좋음 | `<= 0` |
| `complaintControl` | 민원관리지수 | 70 | 높을수록 좋음 | `<= 0` |
| `riskControl` | 폭탄관리지수 | 70 | 높을수록 좋음 | `<= 0` |

모든 지표는 **0~100 범위**로 정규화한다.

### 2.2 효과 적용 + 게임오버 + clamp 순서 (D3 — 단일 표준)
한 선택의 effect를 적용할 때 **반드시 이 순서**를 따른다:

1. **raw 가산**: `raw = stat + effect` (clamp 전 원본값).
2. **게임오버 판정**: raw 기준으로 `checkGameOver(raw)` 평가.
3. **생존 시에만 clamp**: 모든 지표를 `[0, 100]`으로 clamp.
4. 다음 턴으로.

- clamp는 **`applyEffects` 내부 한 곳에서만** 수행한다. StatBar·점수계산·reducer 어디에서도 재clamp 금지.
- 이유: clamp를 먼저 하면 `mental: 1 → -30` 같은 명백한 방전이 `0`에서 멈춰 게임오버를 놓친다. 또 clamp가 없으면 지표가 100을 초과(예: `complaintControl`·`riskControl`)해 점수 인플레이션 버그가 난다.

### 2.3 게임오버 판정 (확정)
- **판정 시점**: 10턴째를 포함한 **모든 턴**의 effect 적용 직후 즉시. 마지막 턴 선택으로 실패조건에 도달하면 등급/엔딩이 아니라 **조기 게임오버**를 표시한다.
- **동시 실패 우선순위** (한 선택으로 두 지표가 동시에 실패조건 도달 시): `mental > progress > complaintControl > riskControl` 순서로 단일 enum 반환.
- 경계 통일: `mental<=0`, `progress<=0`, `complaintControl<=0`, `riskControl<=0` (네 지표 모두 `<=0`으로 통일).

```
type GameOverReason = 'MENTAL_OUT' | 'PROGRESS_OUT' | 'COMPLAINT_OUT' | 'RISK_CONTROL_OUT' | null;
```

## 3. 점수·등급·엔딩 규칙 (확정)

### 3.1 점수식 (D1 — 단순 합산 유지)
```
최종점수 = mental + progress + riskControl + complaintControl   // 네 지표 단순 합산, 이론범위 0~400, 가중치 전부 1.0
```
네 지표가 모두 "높을수록 좋음"으로 통일되어 점수식이 단순 합산이 된다. clamp가 적용되므로 각 항은 0~100, 점수는 0~400로 고정된다. (기존 `(100 - complaint)` 항이 사라지고 `+ complaintControl`로 대체됨)

### 3.2 등급 = 점수 컷 + 게이팅 (D1, D5)
점수만으로 등급을 주지 않는다. **고등급은 4지표가 모두 안전권일 때만** 부여하여 "균형을 맞춰야 고득점"이라는 게임 의도를 강제한다.

게이팅을 만족하지 못하면 **점수가 충족해도 한 단계 강등**한다 (예: 점수는 S인데 한 지표가 위험구간이면 A).

**잠정값** (D5 — 카드 10장 확정 후 시뮬 1000회로 재튜닝):

| 등급 | 점수 컷 | 추가 게이팅(안전권) |
|---|---:|---|
| S | ≥ 320 | 멘탈≥50 ∧ 진척≥50 ∧ 폭탄≥50 ∧ 민원관리≥50 |
| A | ≥ 270 | 멘탈≥35 ∧ 진척≥35 ∧ 폭탄≥35 ∧ 민원관리≥35 |
| B | ≥ 220 | 게이팅 없음 |
| C | ≥ 170 | 게이팅 없음 |
| D | < 170 | — |

> 시작 베이스라인(모든 변동 0) = `70(멘탈)+40(진척)+70(민원관리)+70(폭탄) = 250`점 = B. 이는 "아무것도 안 하면 B"라는 설계 기준선이며, S/A는 균형을 의도적으로 관리해야 도달한다.

### 3.3 특수엔딩 (D4 — 우선순위 확정)
- 특수엔딩은 일반 등급보다 **항상 우선**한다. **단, 게임오버 판이면 특수엔딩을 평가하지 않는다.**
- 조건을 만족하는 **첫 1개만** 적용 (다중 충족 시 아래 선언 순서):

```
SPECIAL_ENDINGS 우선순위 (선언 순서 = 우선순위, first-match):
1. 폭탄돌리기형   : 진척 >= 90 ∧ 폭탄 <= 30
2. 원칙주의형     : 폭탄 >= 85 ∧ 민원관리 <= 30   // 민원관리 낮음 = 게시판이 뜨겁다(기존 민원온도>=70에 해당)
3. 친절과로형     : 민원관리 > 70 ∧ 멘탈 <= 20    // 민원관리 높음 = 모두에게 친절(기존 민원온도<30)
4. 민원관리형     : 민원관리 > 70 ∧ 진척 <= 50
```
- `민원관리 > 70` 공유 충돌은 위 순서로 해소된다(친절과로형이 민원관리형보다 우선).
- 친절과로형(멘탈≤20)과 게임오버(멘탈≤0)는 인접하므로 같은 `resolveEnding`에서 처리: 게임오버면 특수엔딩 평가 자체를 건너뛴다.

### 3.4 엔딩 화면 표기
- 최종점수 표시: `280 / 400`.
- **다음 등급까지 거리**: "A까지 -10점".
- 특수엔딩 시 배지 + "특수엔딩 우선 적용" 표기.
- 등급 한줄평에 **지배 지표 수식어** 1줄 변주(could): 멘탈 최고→"표정은 멀쩡했다", 진척 최고→"일만큼은 다 끝냈다" 등.

## 4. 선택지 설계 원칙 v2 (확정)

### 4.1 트레이드오프 강제 (지배적 선택 제거)
카드 작성 체크리스트:
- 한 카드 안 선택지들의 **net 점수차 ≤ 5** (명백한 정답 방지).
- 각 선택지는 **4지표 중 최소 2개가 반대 부호** (무엇을 포기할지로만 차별화).
- 3지선다의 C(중도)는 절대 net 최댓값이 되지 않게 (셋 중 꼴찌 또는 동률).
- 단일 지표 최대 변동폭 **±25 이내**, 데이터 무결성 테스트 상한은 ±30.

### 4.2 4원형 성립 (D2 — 즉시 비용)
후속이벤트 없이 4원형이 모두 의미를 갖도록 **즉시 비용**으로 설계:

| 원형 | 즉시 효과 패턴 |
|---|---|
| 원칙적 | 폭탄↑, 민원관리↓ 가능, 진척↓/멘탈↓ 가능 |
| 편의적 | 진척↑, 민원관리↑ 가능, **폭탄을 즉시 크게 차감(예 -25→-35)** — "지금 조용하지만 함정" |
| 친절한 설명 | 민원관리↑, 폭탄↑, **멘탈을 크게 차감** |
| 보류 | 멘탈 소모 적음, **진척을 확실히 손실** + 민원관리 소폭↓ 가능 — 재등장 없이도 "미룬 비용"을 즉시 체감 |

- 4원형이 10장에서 **각 최소 2~3회** 등장하도록 매트릭스로 점검.
- **멘탈 회복 밸브**: 멘탈 순증 선택지를 10장 중 2~3개 배치(예: 점심 산책 멘탈+15/진척-5).
- 선택지에 `archetype` 필드를 부여(§7) → 엔딩 플레이스타일 한줄평 집계에 사용.

### 4.3 선택지 개수
- 2지선다 4~5장(날카로운 양자 딜레마) + 3지선다 5~6장. PRD에 표로 고정.
- UI는 2/3개 모두 동일한 세로 스택 그리드 → 레이아웃 흔들림 방지.

## 5. 카드 콘텐츠 (슬롯 정의 — 수치는 D8 별도 작업)

### 5.1 카드 10장 구성 가이드
- 카테고리 분산: 근태 ≤3 / 윤리 3~4 / 행정 2.
- 화자 분산: 팀장 / 직원 / 타부서 / 게시판.
- 미설계 7장 후보 골격: 겸직 허가 / 연차 취소 / 가족 채용 / 재택 근태 / 퇴직자 사적접촉 / 교육자료 긴급수정 / 마지막 결재(10턴 마무리).
- 1턴·10턴을 "퇴근 직전"으로 수미상관 배치해 "무사퇴근" 완결감.

### 5.2 난이도 3막 곡선
- 1막(1~3턴): 중간 변동·튜토리얼. **progress 큰 음수 선택지 배치 금지**(초반 운빨 즉사 방지).
- 2막(4~7턴): 강한 딜레마 + 회복 완충 카드 교차.
- 3막(8~10턴): 변동폭 최대, 엔딩 분기 결정.

### 5.3 결과 문구 톤 가이드
1. 판정이 아닌 **장면**으로: "폭탄관리 -20" (X) → "미래의 내가 조금 위험해졌습니다" (O).
2. "~했지만 ~" **반전 구조**.
3. 직장인 디테일(캡처/게시판/퇴근버튼/샌드위치) 1개 이상.
4. 과장 1문장 1회, 신파·느낌표 남발 금지.
5. **문구에 숫자 언급 금지** (UI가 표시). 좋은예/나쁜예 표로 라이터 가이드화.

### 5.4 페이싱
- 텍스트 상한: 상황 2~3줄 / 선택지 1줄 / 결과 1~2줄.
- 게이팅+트윈 총 지연 카드당 1초 이내.
- 10장 확정 후 분당 500~600자 가정으로 1회 플레이 시간이 3~5분 내인지 검증.

## 6. 데이터 스키마 v2 (확정)

문구는 `src/data/cards.ts`에 한국어 인라인, `satisfies Card[]`로 타입 고정(오타 컴파일 타임 차단).

```ts
type StatKey = 'mental' | 'progress' | 'complaintControl' | 'riskControl';
type Effects = Partial<Record<StatKey, number>>;   // 0/누락 = 변화 없음

type Archetype = 'principled' | 'convenient' | 'kind' | 'defer';

interface Choice {
  id: string;            // 안정 id, 예 'card_001_a' (인덱스 의존 제거)
  text: string;
  archetype: Archetype;
  effects: Effects;
  resultText: string;    // 숫자 없는 장면 문구
}

interface Card {
  id: string;
  category: 'attendance' | 'ethics' | 'admin';
  title: string;
  description: string;
  choices: Choice[];     // 2~3개
}
```

- 지표 메타는 중앙화: `STAT_META: Record<StatKey, { label: string; higherIsBetter: boolean }>` → StatBar가 이 맵으로 4지표를 렌더(중복 마크업 제거).
- `INITIAL_STATS` 단일 상수.
- 새로고침 정책(D6): MVP는 `sessionStorage` 미사용, 새로고침 시 처음부터.

## 7. 아키텍처 (확정 방향)

순수 게임 로직을 UI와 분리한 **`src/game/` 단일 모듈**(폴더명 혼용 금지):

```
src/game/types.ts     : StatKey, Stats, Effects, INITIAL_STATS, STAT_META
src/game/logic.ts     : applyEffects(+clamp), checkGameOver, calculateFinalScore,
                        getGrade(게이팅 포함), resolveEnding (SPECIAL_ENDINGS first-match)
src/game/reducer.ts   : GameState + reducer (START/CHOOSE/NEXT/RESTART)
src/data/cards.ts     : 카드 데이터
src/components/...     : StatBar, Card, ChoiceButton, ResultPanel,
                        StartScreen, GameOverScreen, EndingScreen
```

- **상태머신**: `GameState { phase, turn, stats, lastChoice, gameOverReason, result }` + 단일 reducer로 잘못된 전이 차단.
- **변동값 숨김 계약화**: `ChoiceButton`은 타입상 `effects`를 prop으로 받지 못하게 한다 → "선택 전 변동값 숨김"을 컴파일 타임에 강제.
- **delta 단일 계산**: `CHOOSE` 시 before/after 차이(실적용 delta)를 `state.lastChoice`에 저장 → ResultPanel/StatBar는 그것만 렌더(clamp로 명목값과 실적용값이 갈리는 표시·판정 불일치 차단). `NEXT` 전이 시 `lastChoice` 초기화(stale reveal 방지).
- **다시하기**: `INITIAL_STATE` 단일 상수로 한 번에 리셋(부분 초기화 금지).
- 상태관리 라이브러리 도입 금지 — Context + useReducer.

## 8. UX / 접근성 최소 기준 (확정)

### 8.1 지표 UI
- 네 지표 모두 "높을수록 좋음"으로 통일 → 게이지 찰수록 초록(역방향 지표 없음, 온도계 메타포 폐기).
- 라벨 옆 ↑좋음 표기. **색에만 의존 금지** — 부호 텍스트(`+20`/`-5`/`변화 없음`)와 화살표 항상 병기(색맹 대응).
- 위험구간 경고: 어느 지표든 ≤20이면 게이지 빨강 테두리 + 은은한 맥동(지표당 1개로 제한).
- 선택 후 게임필: +초록/-빨강, 게이지 0.3~0.5초 슬라이드 트윈, floating number.

### 8.2 화면 명세 보강
- **시작 화면**: 게임명, 콘셉트 1~2줄, 4지표 미니 범례(방향 표기), 시작 버튼. (별도 튜토리얼을 범례로 대체)
- **게임오버 화면**: 유형 제목+유머문구, 무너진 지표 빨강 하이라이트, 도달 턴 수, 4지표 스냅샷, 다시하기.
- **엔딩 화면**: §3.4 + 다시하기(게임오버와 동일 위치).
- **진행 표시**: "3/10" 텍스트 + 10칸 스텝 진행바(could).

### 8.3 접근성 (최소선, 풀 WCAG 감사는 향후 제외)
- 선택지/다음/다시하기는 반드시 `<button>` → Tab·Enter/Space 동작, `:focus-visible` 포커스 링 유지(`focus:outline-none` 남발 금지).
- 결과 영역 `aria-live="polite"`, 게임오버 `role="alert"`. 화면 전환 시 새 화면 제목/버튼으로 focus 이동.
- 선택지 버튼 min-height + 줄바꿈(잘림 금지), delta는 `tabular-nums`, 결과 영역 자리 예약으로 layout shift 방지.

### 8.4 반응형
- **mobile-first 단일 컬럼**(진행바+지표 스트립 → 카드 → 선택지 풀폭 세로 스택), 데스크톱만 breakpoint 1개(768px)로 확장.
- 선택지 최소 44px 터치 타깃. 4분할 절대좌표 레이아웃 금지(모바일 붕괴).

### 8.5 공유 (최소선)
- 결과 텍스트 1줄을 `navigator.clipboard`로 복사하는 버튼(~15줄). 이미지·시드 공유 제외.
- 리플레이 변주는 선택지 버튼 **위치 셔플**만(거의 0비용).
